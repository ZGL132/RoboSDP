# RoboSDP 运动学全新数据逻辑说明 (Easy-to-Understand Guide)

## 一、 核心思想：确立“正主”与“参考”

忘掉以前“URDF 主模型”和“DH 主模型”可以随意互相切换的概念。现在的软件逻辑遵循一个铁律：

> **RoboSDP 是参数化机器人设计软件。**
> **DH/MDH 参数表** 是你亲手捏的“橡皮泥”（可塑、可算、可控）。
> **外部 URDF** 只是别人拍的一张“照片”（只能看，不能直接捏）。

*   **内部设计真源 = DH/MDH 参数**：系统里所有正经的物理结构设计，都在这里发生。
*   **外部工程参考 = 导入的 URDF**：仅用于 3D 视图里的“全息投影”参考，以及尝试抄写一份“草稿”。
*   **导出交换格式 = 派生的 URDF**：当你用 DH 设计完毕后，系统替你写出的一份标准文件，交接给下游。

---

## 二、 核心数据流转图 (两根互不干扰的管线)

我们把系统拆成了两条明确的管线，它们在底层计算（Pinocchio引擎）和 3D 显示时是各自独立跑的，**绝不强行互相转换**。

```mermaid
flowchart TD
    %% 管线一
    subgraph 管线1 ["🔵 主线：DH 参数化设计 (设计真源)"]
        DH_UI["UI: DH/MDH 参数表\n(可自由编辑)"] 
        DH_UI -->|代码转换公式| P_API["Pinocchio: API 组装模型"]
        P_API --> FK1["极速 FK 计算矩阵"]
        FK1 --> V1["3D 视图: 绿色骨架"]
        DH_UI -.->|保存时单向输出| URDF_OUT["硬盘: 导出的最小 URDF\n(给下游用的交换文件)"]
    end

    %% 管线二
    subgraph 管线2 ["🟡 辅线：外部 URDF 参考 (外部真源)"]
        URDF_IN["外部导入的 URDF 文件\n(含 Mesh 贴图等)"]
        URDF_IN -->|解析器直接读取| P_Parse["Pinocchio: URDF 原生解析"]
        P_Parse --> FK2["极速 FK 计算矩阵"]
        FK2 --> V2["3D 视图: 真实的 3D 模型"]
        
        URDF_IN -.->|仅作诊断提取| EXTRACT["提取器: 尝试逆推结构"]
        EXTRACT -.-> DRAFT["只读的 DH 草案表格\n(UI 标黄警告: 不可编辑)"]
    end

    %% 唯一桥梁
    DRAFT ===>|用户点击:\n[复制 URDF 草案为 DH 模型]| DH_UI
```

## 三、 两大工作流的真实场景拆解

### 场景 A：从零设计一台机器人（走管线 1 - 🔵 纯正 DH 血统）
1. **新建**：用户从 Topology 拓扑模板选了一个“6R 机械臂”。
2. **编辑**：UI 上的 DH 表格亮起。用户修改了连杆长度 `a` 和 `d`。
3. **动画**：用户拖动关节角滑块。系统把 DH 参数丢给计算引擎（Pinocchio），算出门轴怎么转，3D 视图里的**绿色骨架**实时丝滑联动。
4. **保存**：点击保存。系统把 DH 参数存入项目，并顺手在 `derived/` 目录下生成一个 `xxx.urdf` 文件。**（注意：此时你就算删了那个生成的 URDF，也不影响软件继续设计，因为 DH 才是真源）**。

### 场景 B：导入别人发来的真实机器人 URDF（走管线 2 - 🟡 外部参考血统）
1. **导入**：用户点击 `[导入工程 URDF]`，选择了一个带外壳渲染的 `.urdf` 文件。
2. **显示**：系统**直接把文件**喂给计算引擎（完全不经过 DH 表）。3D 视图里出现了一个逼真的机器人。用户拖动滑块，机器人照样丝滑运动。
3. **诊断草案**：系统在后台偷偷算了一下，在 UI 右侧贴出一张 **黄色的 DH 表（标明：只读诊断草案）**。
    * *情况 1*：系统发现这个 URDF 关节奇形怪状，右上角显示 `诊断级别：无法转换`，复制按钮被锁死。
    * *情况 2*：系统发现这正好是个标准的串联臂，右上角显示 `诊断级别：完美提取`。
4. **过户（核心动作）**：用户看着这张黄色草案觉得没问题，点击了 **`[复制 URDF 草案为 DH 模型]`** 按钮。
5. **重生**：此时，管线 2 结束。系统把这份草案里的数字，硬拷贝到了一份全新的设计真源里。系统抛弃外部 URDF，切换回【场景 A】的管线 1。视图变回绿色骨架，DH 表格解锁，开始真正的参数化设计。

## 四、 Q&A（防钻牛角尖指南）

**Q1：我导入了 URDF，能不能直接在 UI 上改那个黄色只读 DH 表里的数字，然后让 URDF 里的手臂跟着变长？**
**答：绝对不行。** 
URDF 只是个参考底图。你想改数字，必须先点击 `[复制 URDF 草案为 DH 模型]`，把它变成受系统保护的参数化模型（绿色骨架）。此时你改的是新模型的数字，系统会在保存时重新派生一个新的 URDF，而**绝不会去修改你最初导入的那个外部 URDF 源文件**。

**Q2：如果我不把 URDF 复制为 DH 模型，我还能拖动滑块看它运动吗？能用它做正逆解计算吗？**
**答：完全可以。** 
这就是底层计算引擎（Pinocchio）强大的地方。它自带两条路（参考上面的架构图），它直接读取 URDF 就能计算正运动学（FK），拖动滑块极其流畅。但因为它不是参数化模型（没有 DH 表），所以你**不能调整它的连杆长度等固有设计参数**。

**Q3：以前代码里那些复杂的 `master_model_type` 等一堆字段，现在谁来管？**
**答：UI 层再也不用管了。**
开发人员新增了一个 `KinematicModelStatePolicy`（状态策略机）。UI 只要喊一句“我要把这个草案过户成 DH 模型！”，策略机就会在底层把所有字段（是否可编辑、模式名字、引用路径）一次性全部设置正确，彻底杜绝“改漏字段导致保存崩溃”的低级 Bug。
