## 1. 问题描述

### 1.1 现状

当前界面布局如下：

```
┌─────────────────────────────────────────────────────────────────┐
│  Ribbon：7 个标签页（文件 | 建模 | 动力学 | 驱动选型 | 规划与分析 | 结果与导出 | 视图）  │
├──────────┬──────────────────────────────────────┬──────────────┤
│ 项目树   │         中央 3D 视图                  │ 属性面板     │
│ (QTree)  │         (VTK)                        │ (QStacked)   │
│          │                                      │              │
│ 7 个节点 │                                      │ 7 个完整页面 │
│ = 导航    │                                      │ = 编辑器     │
├──────────┴──────────────────────────────────────┴──────────────┤
│  底部日志面板                                                       │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 功能重叠明细

通过代码审查，发现以下具体的功能重叠：

| 操作 | Ribbon 入口 | 项目树入口 | 问题 |
|------|------------|-----------|------|
| 切换到需求页面 | "建模"标签 → "需求定义"按钮 | 点击"任务需求"节点 | 完全相同 |
| 切换到构型页面 | "建模"标签 → "构型设计"按钮 | 点击"构型设计"节点 | 完全相同 |
| 切换到运动学页面 | "建模"标签 → "DH/MDH 建模" / "导入 URDF" | 点击"运动学分析"节点 | 完全相同，且 2 个按钮进同一页面 |
| 切换到动力学页面 | "动力学"标签 → "动力学分析" | 点击"动力学分析"节点 | 完全相同 |
| 切换到规划页面 | "规划与分析"标签 → "规划场景" / "路径规划" / "碰撞检测" | 点击"规划与分析"节点 | 完全相同，且 3 个按钮进同一页面 |
| 切换到方案页面 | "结果与导出"标签 → "方案快照" / "导出结果" | 点击"结果与导出"节点 | 完全相同 |

### 1.3 核心问题

1. **Ribbon 的导航功能完全冗余**：12 个导航按钮与项目树的 7 个节点功能 100% 重叠，用户可以通过两种方式做同一件事，但没有一种方式提供对方无法提供的价值。

2. **Ribbon 存在"空承诺"按钮**："规划与分析"标签中的"规划场景"、"路径规划"、"碰撞检测"三个按钮全部跳转到同一个 Planning 页面，用户点击"碰撞检测"预期看到碰撞检测结果，实际只跳转到页面，仍需手动操作。这是 UX 上的失信行为。

3. **Ribbon 存在禁用按钮浪费空间**："驱动选型"标签中 3 个按钮全部禁用，占用了 Ribbon 空间但没有任何可用功能。

4. **项目树缺少层级**：当前只有一级节点（7 个模块），无法表达模块内部的子功能结构。例如需求模块下应有"基本信息"、"关键工位"、"负载参数"等子区域。

5. **右侧属性面板实际是页面切换器**：[`MainWindow`](apps/desktop-qt/MainWindow.h:164) 的 `m_propertyStack` 是 `QStackedWidget`，右侧面板承载的是完整的模块功能页面，而非真正的属性编辑器。

## 2. 参考成熟软件分析

### 2.1 SolidWorks（最佳参考）

SolidWorks 的界面布局是三面板分工的典范：

```
┌──────────────────────────────────────────────────────────────┐
│  CommandManager (Ribbon) = 特征/草图/装配的命令按钮集合        │
│  ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐  │
│  │ 特征 │ 草图 │ 评估 │ DimXpert│ Office│ 插件 │...  │      │  │
│  └──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘  │
├──────────────┬──────────────────────────────┬────────────────┤
│ FeatureManager│       3D 视图               │ PropertyManager│
│ Tree         │                              │                │
│ (结构浏览)    │                              │ (属性编辑)      │
│              │                              │                │
│ ├── 零件     │                              │ 选中特征的     │
│ │  ├── 草图  │                              │ 参数编辑面板   │
│ │  ├── 凸台  │                              │                │
│ │  ├── 切除  │                              │                │
│ │  └── 阵列  │                              │                │
└──────────────┴──────────────────────────────┴────────────────┘
```

**核心分工原则**：

| 面板 | 角色 | 职责 | 类比 |
|------|------|------|------|
| **CommandManager** (Ribbon) | **命令执行器** | 提供当前模式下的操作按钮（创建特征、编辑草图、标注等） | 工具条 |
| **FeatureManager Tree** (左侧树) | **结构浏览器** | 展示模型的特征层次，选中对象以查看/编辑属性 | 导航 + 选择 |
| **PropertyManager** (右侧面板) | **属性编辑器** | 显示当前选中特征的详细参数，供用户修改 | 属性表单 |

**关键洞察**：SolidWorks 的 Ribbon **从不用于导航**，它的每个按钮都直接执行一个**操作命令**（"创建拉伸"、"添加基准面"、"测量距离"）。导航完全由左侧 FeatureManager Tree 完成。

### 2.2 Qt Creator

Qt Creator 使用更简洁的模式：

```
┌──────────────────────────────────────────────────────────────┐
│  MenuBar: 文件 | 编辑 | 构建 | 调试 | 工具 | 控件 | 帮助      │
├──────────┬──────────────────────────────────────┬────────────┤
│ 模式选择器│         编辑区                        │ 输出面板    │
│ (左侧)   │                                      │            │
│          │                                      │            │
│ ◉ 编辑   │                                      │            │
│ ○ 设计   │                                      │            │
│ ○ Debug  │                                      │            │
│ ○ 项目   │                                      │            │
│ ○ 帮助   │                                      │            │
├──────────┴──────────────────────────────────────┴────────────┤
│  问题 / 搜索结果 / 应用程序输出                                 │
└──────────────────────────────────────────────────────────────┘
```

**核心分工**：
- **模式选择器**（左侧窄条）= 切换工作模式（编辑/设计/调试）
- **项目树**（可集成在编辑区左侧）= 文件结构浏览
- **MenuBar** = 命令菜单

### 2.3 参考总结

| 成熟软件 | Ribbon 的职责 | 树的职责 | 右侧面板的职责 |
|----------|--------------|----------|---------------|
| SolidWorks | 当前模式下的**操作命令** | 特征结构的**层级浏览** | 选中对象的**属性编辑** |
| CATIA | 当前工作台的**命令集合** | 模型结构的**树状浏览** | 选中对象的**参数编辑** |
| Inventor | 当前环境的**工具面板** | 浏览器/模型树 | 属性/参数编辑 |
| VS / VS Code | 菜单/命令面板 = **操作入口** | 解决方案/文件**结构树** | 选中文件的**属性窗口** |

**共同规律**：
- Ribbon/MenuBar = **执行操作**（"做什么"）
- Tree = **浏览结构**（"看什么"）
- Property Panel = **编辑参数**（"改什么"）

## 3. 优化目标与原则

### 3.1 优化目标

1. **消除功能重叠**：Ribbon 不再承载页面导航功能，专注操作命令
2. **消除空承诺按钮**：每个 Ribbon 按钮必须有明确的、即时执行的操作
3. **消除禁用按钮**：未实现的按钮不占用 Ribbon 空间
4. **项目树增加层级**：支持模块内部的子功能导航
5. **三面板各司其职**：Ribbon=命令、树=导航、属性面板=编辑

### 3.2 优化原则

1. **向后兼容**：不破坏已有信号/槽接口，只改造 Ribbon 和 MainWindow 的布局逻辑
2. **渐进实施**：分 3 步完成，每一步独立可编译可运行
3. **可验证**：每步完成后用户可直接验收
4. **参考成熟模式**：以 SolidWorks 的 CommandManager + FeatureManager Tree + PropertyManager 为蓝本

## 4. 方案设计：角色重定义

### 4.1 新角色定义

| 面板 | 旧角色 | 新角色 | 核心变化 |
|------|--------|--------|----------|
| **Ribbon** | 导航 + 视图开关 | **上下文命令面板** | 移除所有导航按钮，改为执行按钮 |
| **项目树** | 一级导航 | **层级结构浏览器 + 导航** | 增加子节点，支持模块内导航 |
| **右侧属性面板** | 页面容器（QStackedWidget） | **真正的属性/编辑面板** | 从承载完整页面变为承载属性编辑器 |
| **中央区域** | 仅 VTK 视图 | **VTK 视图 + 嵌入编辑区** | 复杂编辑页可嵌入中央区域 |

### 4.2 新旧对比

**旧模式**：
```
Ribbon ──── 导航 ────→ 切换项目树选中节点 ────→ 切换右侧面板页面
项目树 ──── 点击节点 ─→ 切换右侧面板页面
右侧面板 ── 承载完整功能页面（7 个 Widget 放在 QStackedWidget 中）
```

**新模式**：
```
Ribbon ──── 执行命令 ──→ 触发计算/生成/验证/保存等操作
项目树 ──── 选择节点 ──→ 模块级导航 + 子功能定位
右侧面板 ── 属性编辑 ──→ 显示当前选中对象的可编辑属性
中央区域 ── 显示页面 ──→ 显示当前模块的完整编辑页面 + VTK 视图
```

## 5. Ribbon 改造方案

### 5.1 标签结构重新设计

从 **7 个标签** 精简为 **4 个标签**，消除冗余和禁用按钮：

| 新标签 | 旧标签 | 变化说明 |
|--------|--------|----------|
| [`文件`](#51-文件标签) | 文件 | 保持不变，功能合理 |
| [`建模`](#52-建模标签) | 建模 + 动力学 | 合并，按钮根据当前模块动态切换上下文 |
| [`分析`](#53-分析标签) | 驱动选型 + 规划与分析 + 结果与导出 | 合并，按钮根据当前模块动态切换上下文 |
| [`视图`](#54-视图标签) | 视图 | 保持不变，功能合理 |

### 5.2 文件标签

保持不变，始终可用。

| 按钮 | 功能 | 说明 |
|------|------|------|
| 新建项目 | 创建新项目 | ✅ 已有实现 |
| 打开项目 | 打开已有项目 | ✅ 已有实现 |
| 保存 | 全局保存 | ✅ 已有实现 |
| 导出 JSON | 导出方案 | ❌ 原为禁用，改为可用后跳转到 Scheme 的导出子功能 |

### 5.3 建模标签

**核心改动**：按钮不再用于导航，而是根据当前激活的模块**动态显示对应的操作命令**。

#### 当需求模块激活时：
| 按钮 | 功能 | 说明 |
|------|------|------|
| + 添加工位 | 在需求中添加一个关键工位 | 直接打开工位编辑对话框 |
| + 添加工况 | 添加负载工况变体 | 直接打开工况编辑对话框 |
| ✅ 验证需求 | 执行需求校验 | 调用 [`RequirementValidator`](modules/requirement/validator/RequirementValidator.h) |

#### 当构型模块激活时：
| 按钮 | 功能 | 说明 |
|------|------|------|
| 🔄 刷新模板 | 从模板目录重新加载模板列表 | 调用 [`TopologyTemplateLoader`](modules/topology/service/TopologyTemplateLoader.h) |
| ⚡ 生成候选 | 根据当前参数生成候选构型 | 调用 [`TopologyService`](modules/topology/service/TopologyService.h) |
| ✅ 验证构型 | 执行构型校验 | 调用 [`TopologyValidator`](modules/topology/validator/TopologyValidator.h) |
| 💾 保存草稿 | 保存当前构型草稿 | 调用 `SaveCurrentDraft()` |

#### 当运动学模块激活时：
| 按钮 | 功能 | 说明 |
|------|------|------|
| ▶ FK 求解 | 执行正向运动学计算 | 调用 `KinematicsService.BuildFK()` |
| 🔄 IK 求解 | 执行逆运动学求解 | 调用 `KinematicsService.SolveIK()` |
| 🌐 工作空间采样 | 执行工作空间蒙特卡洛采样 | 调用 `KinematicsService.SampleWorkspace()` |
| 📥 导入 URDF | 从文件导入 URDF 模型 | 调用 URDF 导入对话框 |

### 5.6 Ribbon 上下文切换机制

```cpp
// 伪代码：Ribbon 上下文切换逻辑
void RibbonBarWidget::SetActiveModule(ModuleType module)
{
    // 1. 清除"建模"标签中的旧按钮
    ClearModelingTabButtons();
    
    // 2. 根据当前模块动态添加新按钮
    switch (module)
    {
        case ModuleType::Requirement:
            AddModelingButton("+ 添加工位", &signalAddKeyPose);
            AddModelingButton("+ 添加工况", &signalAddLoadVariant);
            AddModelingButton("✅ 验证需求", &signalValidateRequirement);
            break;
        case ModuleType::Topology:
            AddModelingButton("🔄 刷新模板", &signalRefreshTemplates);
            AddModelingButton("⚡ 生成候选", &signalGenerateCandidates);
            AddModelingButton("✅ 验证构型", &signalValidateTopology);
            AddModelingButton("💾 保存草稿", &signalSaveTopologyDraft);
            break;
        case ModuleType::Kinematics:
            // ...
    }
    
    // 3. 切换到"建模"标签
    m_tabs->setCurrentIndex(1);
}
```

Ribbon 通过 `signalActiveModuleChanged(ModuleType)` 信号与 MainWindow 联动。当 MainWindow 切换模块时，通知 Ribbon 更新上下文按钮。

### 6.2 项目树节点类型定义

```cpp
enum class TreeNodeType {
    // 模块级节点
    ProjectRoot,
    Requirement,
    Topology,
    Kinematics,
```

## 7. 右侧属性面板改造方案

### 7.1 当前问题

右侧 [`m_propertyStack`](apps/desktop-qt/MainWindow.h:164) 是 `QStackedWidget`，承载的是**完整的 7 个模块 Widget**（每个 Widget 数百行代码），而非属性编辑器。

```
右侧面板现状（QStackedWidget）：
┌─────────────────────┐
│ RequirementWidget   │  ← 完整的功能页面（数百行）
│ TopologyWidget      │  ← 完整的功能页面（数百行）
│ KinematicsWidget    │  ← 完整的功能页面（数百行）
│ DynamicsWidget      │  ← 完整的功能页面（数百行）
│ ...                 │
└─────────────────────┘
```

### 7.2 改造方案

将右侧面板拆分为**真正的属性编辑器**，遵循 SolidWorks PropertyManager 模式。

#### 第一步（短期）：属性树 + 操作按钮

```
右侧面板改造后：
┌─────────────────────┐
│ 属性面板             │
├─────────────────────┤
│ 🔍 搜索属性...       │  ← 搜索框
├─────────────────────┤
│ ▼ 基本信息           │  ← 可折叠 Section
│ ├ 项目名称: [____]   │
│ ├ 场景类型: [▼]     │
│ └ 描述:    [____]   │
├─────────────────────┤
│ ▼ 关键参数           │  ← 可折叠 Section
│ ├ 额定负载: [__] kg │
│ ├ 最大负载: [__] kg │
│ └ 工具质量: [__] kg │
├─────────────────────┤
│ [应用] [重置]        │  ← 操作按钮
└─────────────────────┘
```

#### 第二步（长期）：Qt 属性浏览器

使用 `QtTreePropertyBrowser` 或 `QTreeView` + 自定义委托实现属性编辑：

```
右侧面板（属性浏览器模式）：
┌─────────────────────┐
│ 属性                 │  值
├─────────────────────┤
│ ▶ 元信息             │
│ ▼ 负载参数           │
│   额定负载           │ 10.0 kg
│   最大负载           │ 15.0 kg
│   工具质量           │ 1.5 kg
│   负载质心 X         │ 0.0 m
│   负载质心 Y         │ 0.0 m
│   负载质心 Z         │ 0.15 m
│ ▶ 工作空间需求        │
│ ▶ 关节限位            │
└─────────────────────┘
```

### 7.3 模块页面迁至中央区域

原来的 7 个模块 Widget（RequirementWidget、TopologyWidget 等）从右侧 `QStackedWidget` 迁出，改为以**浮动面板**或**分栏**形式显示在中央区域（与 VTK 视图并列或通过标签切换）。

```
中央区域改造后：

│                                              │
│  ┌──────────────────┬──────────────────────┐  │
│  │                  │  模块编辑面板          │  │
│  │   3D VTK 视图    │  (原右侧 Widget 迁入) │  │
│  │                  │                      │  │
│  │                  │  DH参数表             │  │
│  │                  │  关节限位             │  │

│  └──────────────────┴──────────────────────┘  │
└──────────────────────────────────────────────┘
```

## 8. 三面板联动流程

### 8.1 用户操作流程示例

以"用户从需求定义进入，编辑关键工位，然后切换到运动学建模"为例：

```
1. 用户点击左侧项目树 "需求定义 → 关键工位"
   ↓
2. 项目树选中 "关键工位" 节点
   ↓
3. MainWindow.HandleProjectTreeSelectionChanged()
   ↓
4. a) 右侧面板 → 显示关键工位属性编辑器
   b) 中央区域 → 显示需求模块页面，聚焦工位列表
   c) VTK 视图 → 显示当前工位的坐标系标记
   d) Ribbon → 切换到"建模"标签，显示需求模块的操作按钮
      [+ 添加工位] [+ 添加工况] [✅ 验证需求]

5. 用户点击 Ribbon 上的 "+ 添加工位"
   ↓
6. Ribbon 发出 signalAddKeyPose 信号
   ↓
7. MainWindow 将信号转发给 RequirementWidget.OnAddKeyPose()
   ↓
8. 添加工位对话框打开，用户输入参数
   ↓
9. 完成后，项目树自动刷新显示新工位节点，VTK 更新显示

10. 用户点击左侧项目树 "运动学建模 → DH/MDH 参数"
    ↓
11. MainWindow 切换上下文：
    a) 右侧面板 → 显示 DH 参数属性编辑器
    b) 中央区域 → 显示运动学模块页面
    c) VTK 视图 → 显示运动学骨架
    d) Ribbon → 切换到运动学操作按钮
       [▶ FK 求解] [🔄 IK 求解] [🌐 工作空间采样] [📥 导入 URDF]
```

### 8.2 信号流设计

```
用户点击项目树节点
       │
       ▼
QTreeWidget::currentItemChanged
       │
       ▼
MainWindow::HandleProjectTreeSelectionChanged(QTreeWidgetItem*)
       │
       ├─→ 更新 Ribbon 上下文（SetActiveModule）
       │       │
       │       └─→ RibbonBarWidget::SetActiveModule(ModuleType)
       │               │
       │               └─→ 动态重建"建模"/"分析"标签的按钮
       │
       ├─→ 更新右侧属性面板
       │       │
       │       └─→ PropertyPanelWidget::SetCurrentNode(TreeNodeType)
       │               │
       │               └─→ 切换属性编辑器内容
       │
       ├─→ 更新中央区域
       │       │
       │       └─→ 切换中央区域显示的模块页面（或保持 VTK + 叠加编辑面板）
       │
       └─→ 更新 VTK 视图状态
               │
               └─→ 显示/隐藏对应模块的 3D 内容
```

## 12. 风险与注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 项目树多层结构可能导致右键菜单逻辑复杂化 | 中 | 右键菜单仅在根节点和模块级节点上触发，子节点暂不添加右键菜单 |
| Ribbon 按钮动态重建可能导致布局闪烁 | 低 | 使用 `QStackedWidget` 预构建各模块的按钮面板，切换时直接切换面板而非逐个添加/移除按钮 |
| 现有导航信号可能被其他模块依赖 | 低 | 保留旧信号但标记为弃用(`[[deprecated]]`)，逐步迁移 |
| 属性面板改为属性编辑器后，复杂编辑操作（如 DH 参数表编辑）的交互可能受限 | 中 | 复杂编辑操作仍在中央区域的模块页面中完成，属性面板仅显示关键参数的快捷编辑 |
| Phase 3 工作量较大 | 中 | Phase 3 可推迟到后续迭代，Phase 1+2 即可独立交付 |
